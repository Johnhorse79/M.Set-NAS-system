Section 4 — Challenges and Troubleshooting
One of the most important lessons from this build is that real infrastructure work involves unexpected failures. Every challenge documented here required methodical diagnosis, research, and iteration to resolve. The troubleshooting process itself is as valuable as the completed build — it demonstrates the same diagnostic thinking used in professional cloud and infrastructure roles.
A secondary lesson: document in real time. Many of the details in this section were difficult to reconstruct after the fact because the focus during the build was on finding solutions rather than recording them. Future projects will include a running build log updated after each session.

4.1 Hardware Challenges
Challenge: No HDMI output on the Precision 3630

Situation: The Precision 3630 has DisplayPort outputs only — no HDMI
Impact: Could not connect to a TV or standard monitor without an adapter
Resolution: Purchased a DisplayPort to HDMI adapter — connected to TV for initial setup
Lesson: Always verify display output compatibility before beginning a build

Challenge: Drive brackets not included

Situation: The WD Red Pro 14TB drives required 3.5" mounting brackets not included with the Precision 3630
Impact: Could not secure drives in bays without additional hardware
Resolution: Purchased aftermarket brackets separately before assembly

Challenge: Boot drive misunderstanding

Situation: Initially removed the 500GB SSD and replaced it with a hard drive — not understanding TrueNAS requires a dedicated boot drive separate from storage drives
Impact: Delayed assembly and required partial disassembly to correct
Resolution: Reinstalled the SSD as the dedicated boot drive and moved both WD Red Pro drives to HDD bays
Lesson: TrueNAS SCALE requires a dedicated boot device — the OS drive must be separate from the ZFS storage pool drives


4.2 Storage Challenges
Challenge: ZFS mirror pool creation failed repeatedly

Situation: Every attempt to create the ZFS mirror pool returned [EFAULT] [Errno 5] Input/output error
Initial diagnosis: Assumed loose SATA cable — reseated all cables and confirmed clicks on both ends
Further diagnosis: Opened Linux Shell and ran dmesg | grep -i error which returned:

ata3.00: error: { UNC }
I/O error, dev sdb, sector 27344764800
Buffer I/O error on dev sdb, logical block 3418095600, async page read

Isolation testing: Created individual stripe pools with each drive separately — both drives worked individually confirming the issue was specific to mirror initialization on sdb
SMART testing:

smartctl -t short /dev/sdb — returned scsi error medium or hardware error (serious)
smartctl -H /dev/sdb — returned PASSED but error log showed 8 hardware errors
smartctl -l error /dev/sdb — confirmed 8 hardware errors with Emask 0x1 failures and read log failed: scsi error medium or hardware error (serious)


Resolution: Confirmed sdb (serial 9RK3622C, model WDC WD140EFFX) is defective — contacted Amazon seller with full diagnostic output requesting replacement. Created single drive stripe pool with healthy sdc as temporary measure
Lesson: Always run SMART diagnostics on new drives before deployment. Test drives individually before attempting RAID/mirror configuration. New drives can fail out of the box — this is called infant mortality failure

Challenge: Mismatched drive models in mirror pair

Situation: During diagnostics discovered the two drives were different models — WDC WD140EFFX (WD Gold, 5400 RPM) and WDC WD141KFGX (WD Red Pro, 7200 RPM)
Impact: Different RPM speeds and firmware versions in a mirror pair — not ideal
Resolution: ZFS mirror works with mismatched drives — performance throttles to the slower drive. Replacement drive will be ordered as WD Red Pro to match the healthy sdc drive
Lesson: Verify drive models match exactly when ordering a mirror pair — same model, same firmware generation

Challenge: SATA port color significance unknown

Situation: Motherboard had one tan/cream colored SATA port among three black ports — purpose unknown
Resolution: Avoided the tan port entirely — all drives connected to black SATA ports only
Lesson: Research motherboard SATA controller specifications before assembly — colored ports often indicate different controllers with different performance or compatibility characteristics


4.3 Nextcloud Challenges
Challenge: First Nextcloud installation — container crashed silently

Situation: Nextcloud showed Running status in TrueNAS Apps but web UI was completely unreachable
Diagnosis: Ran ss -tlnp | grep 30027 — port 30027 not listening. Ran midclt call app.query — confirmed nextcloud container state was exited while postgres container was running
Root cause: Both AppData Storage and User Data Storage were configured as Host Path /mnt/datapool — Nextcloud web files and data directory pointed to the same location causing a conflict on startup
Resolution: Deleted Nextcloud app including ixVolumes and reinstalled with corrected storage configuration:

AppData Storage → ixVolume (TrueNAS managed) — not Host Path
User Data Storage → Host Path /mnt/datapool
Postgres Storage → ixVolume (TrueNAS managed)


Lesson: In TrueNAS SCALE Nextcloud app installations only User Data Storage should point to a Host Path — AppData and Postgres should remain as ixVolume to prevent path conflicts

Challenge: TrueNAS datasets conflicting with Nextcloud user creation

Situation: Created TrueNAS datasets named unit1, unit2, unit3, unit4 before installing Nextcloud. When creating matching Nextcloud user accounts received error: Login is invalid because files already exist for this user
Root cause: Nextcloud found existing folders matching the usernames in /mnt/datapool and refused to create accounts for them since it didn't create those folders itself
Resolution: Stopped Nextcloud, deleted the four conflicting TrueNAS datasets through the web UI, restarted Nextcloud, created user accounts — Nextcloud created the user folders itself
Lesson: When using Nextcloud with a Host Path data directory let Nextcloud create and manage user folders — do not pre-create folders or datasets with names matching intended usernames

Challenge: Untrusted domain error when accessing via Tailscale

Situation: After Tailscale was configured accessing Nextcloud via http://100.76.193.123:30027 showed: Access through untrusted domain
Root cause: Nextcloud only trusted the IP configured during installation — Tailscale IP was not in the trusted_domains list
Resolution: Used docker exec as root to add trusted domains via occ command line tool — added both local IP and Tailscale IP with and without port numbers
Lesson: Add all expected access URLs to trusted_domains during initial Nextcloud configuration — include local IP, Tailscale IP, and both with port numbers

Challenge: Docker permission denied for truenas_admin

Situation: Running docker exec commands as truenas_admin returned permission denied
Resolution: Used sudo su to switch to root before running docker commands
Lesson: TrueNAS restricts docker socket access to root — always switch to root for container management commands


4.4 Network Challenges
Challenge: IP address changed when moving PC between locations

Situation: During build the NAS was moved between locations — DHCP assigned different IP addresses at each location making the system unreachable
Resolution: Set permanent static IP at the permanent location after confirming the network range
Lesson: Set a static IP as one of the first configuration steps after confirming the permanent network location

Challenge: Static IP caused connectivity loss at temporary locations

Situation: After setting static IP 192.168.1.196 the NAS became completely unreachable at non-permanent locations with different network ranges
Resolution: Used TrueNAS Console Setup Menu Linux Shell to temporarily switch to DHCP using midclt commands when access was needed at other locations
Lesson: A static IP is essential for production stability but creates access challenges during a mobile build process — plan the permanent location before setting static networking

Challenge: Remote access not working from outside home network

Situation: Family members attempting to access Nextcloud from different WiFi networks could not reach 192.168.1.196 — connection timed out
Root cause: Local IP addresses are only routable within the local network — unreachable from external networks
Resolution: Tailscale provides a persistent IP (100.76.193.123) accessible from any network — all remote users instructed to use Tailscale IP with Tailscale app installed on their devices
Lesson: Never rely solely on local IP for a multi-user storage system — always deploy a remote access solution before distributing access credentials to users


4.5 rclone / Azure Backup Challenges
Challenge: Permission denied running rclone as truenas_admin

Situation: Running rclone copy /mnt/datapool as truenas_admin returned permission denied — datapool mounted with root-only access
Resolution: Switched to root with sudo su before running rclone

Challenge: rclone config not found when running as root

Situation: Root user has a separate config directory — the azurebackup remote configured as truenas_admin was not visible to root
Resolution: Passed explicit config file path using --config flag:

bashrclone copy /mnt/datapool azurebackup:nasbackup --progress --config /home/truenas_admin/.config/rclone/rclone.conf
Challenge: Azure access key exposed in chat

Situation: During rclone configuration the Azure storage account access key was accidentally pasted into the chat conversation instead of directly into the terminal
Resolution: Immediately rotated Key1 in Azure portal — generating a new key invalidates the exposed key
Lesson: Never paste credentials into chat, email, or any communication platform — always enter sensitive keys directly into the terminal or a password manager


4.6 Documentation Lesson
A recurring theme throughout this build was the tension between solving the problem and documenting the solution. When an error occurred the instinct was to focus entirely on resolution — running commands, trying different approaches, iterating quickly. This is the right instinct for getting the system working but it comes at the cost of documentation quality.
Many of the details in this section were reconstructed after the fact from conversation logs rather than recorded in real time. Key specifics — exact error messages, the sequence of commands tried, what was ruled out before the solution was found — were difficult to recall accurately.
For future builds the following will be implemented:

A running markdown build log updated after each session
Screenshots of error messages captured immediately when they occur
A decision log noting what was tried, what failed, and why the final solution worked
GitHub commits after each major milestone so progress is version controlled in real time

This approach transforms the build process itself into a portfolio artifact — showing not just the final result but the professional discipline of documenting infrastructure work as it happens.
