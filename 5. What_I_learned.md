Section 5 — What I Learned / What I'd Do Differently
5.1 The Most Important Lessons
Document everything in real time.
This is the single most valuable lesson from this build. When a problem was encountered the instinct was always to focus entirely on finding the solution — running commands, trying different approaches, iterating as fast as possible. That instinct is correct for getting the system working. But it comes at a significant cost: by the time the solution was found the exact sequence of what was tried, what failed, and why the final fix worked was already fading.
Reconstructing the troubleshooting log for this README after the fact was significantly harder than it would have been to write one or two sentences after each session. A running build log — even informal notes in a markdown file — would have produced a more accurate and detailed record of the real work done.
This lesson applies directly to professional infrastructure work. In regulated environments like government-funded nonprofits documentation isn't optional — it's an audit requirement. Change logs, incident reports, and remediation records are part of the compliance fabric. This build made that abstract requirement concrete and personal.
IP addresses are the phone numbers of the internet — and they change.
Before this build the concept of IP addresses was understood theoretically. After this build it was understood experientially. The NAS became unreachable multiple times during the build process for a single reason: the IP address changed when the device moved to a different network. Every time a new network assigned a new DHCP address the system effectively disappeared — no one could reach it because the address they knew was no longer valid.
Setting a static IP resolved this permanently. Once 192.168.1.196 was locked to the NAS it became reliably reachable every time — as long as you were on the same network. That led directly to the next lesson: local IPs only work locally, which is exactly why Tailscale was essential.
Real-time troubleshooting made this lesson stick in a way that no tutorial ever could. Reading about static vs DHCP is one thing. Losing access to a system you built because the address changed is another.

5.2 What I Would Do Differently
Match drive models exactly when building a mirror pair.
Both WD Red Pro 14TB drives were ordered at the same time from the same listing — but when diagnostics were run during troubleshooting it was discovered they were actually different models. One was a WD Gold (WDC WD140EFFX, 5400 RPM) and one was a WD Red Pro (WDC WD141KFGX, 7200 RPM). A ZFS mirror works with mismatched drives but it is not ideal — the mirror runs at the speed of the slower drive and different firmware versions can behave differently under stress.
The correct approach is to verify the exact model number before purchasing — not just the marketing name. WD Red Pro and WD Gold are different product lines with different RPM ratings, firmware, and workload specifications even when sold under similar descriptions.
Run SMART diagnostics on new drives before beginning configuration.
One of the WD Red Pro drives was defective out of the box — a condition known as infant mortality failure that affects a small percentage of new drives. Several hours were spent troubleshooting SATA cables, ports, and pool configurations before the diagnostics pointed clearly to the drive itself. Running smartctl -t short and smartctl -l error on both drives before attempting pool creation would have identified the defective drive immediately and saved significant time.
Set the static IP before any application configuration.
Network configuration was treated as a secondary concern during the build — something to handle after the applications were installed. This was a mistake. Because the IP address kept changing between locations, application configurations like Nextcloud's trusted domains had to be updated multiple times. Setting the static IP at the permanent location as the very first post-installation step would have eliminated this entire category of problems.
Document in real time from day one.
If starting over a markdown build log would be created before the first component was installed and updated after every session. Even brief notes — what was attempted, what error appeared, what was tried, what finally worked — would have produced a far more complete and accurate record of the build.

5.3 What Surprised Me
How long real troubleshooting takes.
The full build took approximately 5 days with 3 of those days spent working most of the day. Before starting the expectation was that following a plan would produce a predictable timeline. What the build actually demonstrated is that infrastructure work is non-linear — a single defective component or misconfiguration can consume hours of diagnostic work before the root cause is identified.
The defective hard drive set the tone early. What appeared to be a simple pool creation step turned into a multi-session diagnostic process involving SMART tests, Linux shell commands, port swapping, cable replacement, and isolation testing. That experience recalibrated expectations for the rest of the build — every subsequent step was approached with more patience and less assumption about how long it would take.
How satisfying it is when it works.
After days of troubleshooting the moment when Nextcloud loaded on a phone over cellular data using the Tailscale IP — completely away from the home network — was genuinely significant. It was not a tutorial result. It was a real system built from hardware, configured through real errors and real fixes, working exactly as designed. That experience is different from completing a course or passing a certification exam.

5.4 Professional Connections
Tailscale as enterprise zero-trust networking.
The most directly transferable professional skill from this build is the implementation and understanding of zero-trust remote access. Tailscale uses the same security model that enterprise organizations deploy for remote workforce access — authenticated encrypted tunnels with no open ports exposed to the public internet. Configuring Tailscale on a self-hosted server and connecting multiple devices to it is the same conceptual work as deploying zero-trust network access in a regulated environment.
For government-funded nonprofits managing sensitive client data — where staff work remotely across multiple sites — this model is directly applicable. Case managers accessing systems from community locations, administrators connecting from home, and auditors requiring read-only access to compliance records all represent the same remote access problem that Tailscale solves at scale.
IP address management as infrastructure discipline.
The hands-on experience with DHCP vs static IP, network segmentation between locations, and the practical consequences of address changes translated an abstract networking concept into operational knowledge. In a cloud environment this maps directly to understanding the difference between dynamic and static resource allocation — a core FinOps concept applied to network infrastructure.
Troubleshooting methodology.
The diagnostic process used to identify the defective hard drive — isolating variables, running targeted commands, reading error logs, testing components individually before testing them together — is the same methodology used in cloud infrastructure troubleshooting. The specific tools change (dmesg and smartctl vs Azure Monitor and Log Analytics) but the thinking is identical: reproduce the error, isolate the variable, confirm the fix.
